# SameFile_Finder v2.0.0 設計メモ（Rust版）

## 1. 目的

SameFile_Finder v2.0.0 は、現行の Python 版（v1.x）の思想を維持しつつ、Rust で再構築する次世代版とする。

Rust 化の主目的は以下：

- **配布性の向上**（単体 exe 配布、依存トラブル低減）
- **堅牢性の向上**（NAS/SMB 特有のエラーに強い）
- **設計の明確化**（core / UI / DB 分離、型による整理）

> 重要：v2.0.0 は「速度を劇的に上げる版」ではなく、  
> **v1 の思想を保ったまま、配布性と堅牢性を高める基盤版** とする。

---

## 2. 前提と現状認識

### 2.1 v1.x の現状
- Python + GUI 完結
- 判定ロジックは **サイズ一致で候補化 → ハッシュで確定**
- 実用上かなり安定しており、非破壊・一覧確認の運用に適している

### 2.2 実運用環境
- NAS 越し HDD（SMB 想定）を走査することが多い
- 動画ファイルなど大容量ファイルも対象

### 2.3 ボトルネックの理解
- 多くの場合、ボトルネックは **CPU ではなく I/O（転送・読み込み）**
- そのため、Rust 化のみでの劇的な高速化は期待しにくい
- 体感速度改善には「言語差」より **転送量削減・段階絞り込み設計** が効く

---

## 3. v2.0.0 として Rust 化する理由

### 3.1 配布性（最大のメリット）
- 単体 exe 配布がしやすい
- Python 実行環境・依存関係の差異を吸収しやすい
- 友人や他環境への配布が容易

### 3.2 堅牢性
NAS/SMB 走査では、以下のような事象が起こりやすい：

- 権限エラー
- ファイルロック
- 途中でファイルが消える
- ネットワーク切断
- 長パス問題

これらを「異常終了」ではなく、**スキップ理由付きで観測可能**にする設計を取りやすい。

### 3.3 設計の進化（保守性）
- core（純ロジック）と UI / DB / FS を分離しやすい
- パイプライン設計を明示しやすい
- `Result` / `enum` / 型で曖昧さを潰しやすい

---

## 4. v2.0.0 の基本方針（v1互換の核を維持）

v2.0.0 では、判定ロジックの思想は v1.x と同じものを維持する。

### 判定フロー（v2.0.0）
1. **Stage 1: size で候補化**（必須）
2. **Stage 3: hash で確定**（必須）

※ Stage 2（fingerprint）は v2.1.0 以降で追加予定

### 方針
- まずは **v1互換（size → hash）** を Rust で安定動作させる
- Rust 初学フェーズでは欲張らず、作品として完成させることを優先する

---

## 5. v2.0.0 のスコープ（MVP）

### 5.1 実装対象（入れる）
- Rust 単体 exe（GUI 完結）
- 重複検出ロジック（size → hash）
- 進捗表示
- キャンセル機能
- 重複グループ一覧表示
- グループ展開（同一ハッシュのファイル一覧）
- ダブルクリックで Explorer を開く（Windows）
- エクスポート（CSV）

※ JSON エクスポートは余力があれば追加

### 5.2 実装しない（v2.0.0では見送る）
- fingerprint（部分ハッシュ）
- SQLite キャッシュ
- A/B 比較モード
- ハッシュ方式切替 UI
- 除外フォルダ UI（将来追加候補）

---

## 6. v2.1.0 以降の拡張予定（ロードマップ）

### v2.1.0（NAS向け体感改善）
#### 6.1 fingerprint の導入
- size 一致後、先頭/末尾 N KB などの fingerprint でさらに絞る
- フルハッシュ対象を減らして転送量削減

#### 6.2 SQLite キャッシュ
保存対象（例）：
- `path`
- `size`
- `mtime`
- `fingerprint`
- `hash`

次回走査時に差分だけ再計算できるようにする。

#### 6.3 ハッシュ方式の拡張
- v2.0.0 では v1互換の方式を採用（MD5 or SHA-256）
- 将来的に `blake3` などを追加可能な設計にする

例（将来）：
- `--hash md5`
- `--hash sha256`
- `--hash blake3`

### v2.2.0 以降（機能拡張）
- A/B 比較モード（バックアップ照合）
- 除外フォルダ／拡張子フィルタ
- 設定保存（config）
- ログ表示の強化（スキップ理由の集計など）

---

## 7. NAS 越しで効く設計方針（重要）

Rust 化よりも、以下のような設計の工夫が体感速度に効く。

### 7.1 段階フィルタ（転送量削減）
- size 一致
- （将来）fingerprint 一致
- フルハッシュ

### 7.2 並列しすぎない
HDD + SMB 環境では、並列度を上げると逆に遅くなる場合がある。

方針：
- デフォルトは **低並列（1〜2）**
- 将来設定化可能な設計にする（v2.0.0 は固定でもよい）

### 7.3 キャッシュを将来の主戦略にする
- v2.0.0 時点では未実装でもよい
- v2.1.0 以降での最大改善ポイントとして位置付ける

---

## 8. 設計方針（“裏を覗いても美しい”構成）

## 8.1 レイヤ分離
### core（純ロジック）
- 走査対象の候補化
- サイズグルーピング
- ハッシュ計算（抽象化された Reader 経由も検討）
- 重複グループ確定
- ドメイン型（Result / Event / Config）

### adapters（I/O）
- ファイルシステムアクセス
- CSV 出力
- （将来）SQLite
- Explorer 連携（Windows）
- ログ出力

### ui
- egui ベースの GUI
- Event を購読して描画更新
- ユーザー操作（開始/停止/エクスポート）

### app
- 設定生成
- 依存注入
- 起動処理の結線

---

## 8.2 進捗通知は Event ストリームで行う
処理側が UI を直接触らない。UI はイベントを受け取って表示する。

### Event 例（案）
- `Scanning(path)`
- `SizeGrouped(total_files, candidate_files)`
- `Hashing(path, index, total)`
- `Hashed(path)`
- `Skipped(path, reason)`
- `FoundGroup(hash, count)`
- `Done(summary)`

※ 実際の enum 名は実装時に調整

---

## 8.3 エラーは分類して“スキップ理由”として扱う
落ちるのではなく、観測可能にする。

### 例
- `PermissionDenied`
- `FileNotFound`
- `PathTooLong`
- `Locked`
- `NetworkDisconnected`
- `IoError(String)`

UI やログでは「どのファイルが、なぜスキップされたか」を確認できるようにする。

---

## 9. UI 方針（Rust GUI）

### v2.0.0 候補：egui（第一候補）
理由：
- 純 Rust で完結しやすい
- 単体 exe 配布と相性が良い
- ツール系 UI（テーブル、ボタン、進捗、ログ）に向いている
- Rust 初学でも比較的着手しやすい

### 他候補（将来含む）
- Tauri（見た目の自由度は高いが構成が増える）
- Slint（ネイティブ寄り、学習コスト中）

方針として、v2.0.0 は **egui で素直に作品化** を優先する。

---

## 10. ハッシュ方式方針

### v2.0.0
- **v1.x と同じ方式** を採用する（互換優先）
  - v1 が MD5 なら v2 も MD5
  - v1 が SHA-256 なら v2 も SHA-256

### 将来（v2.1.0+）
- ハッシュ方式を選択可能にできる設計にする
- 候補：MD5 / SHA-256 / BLAKE3

※ 重複検出用途では、改ざん耐性よりも実用上の識別精度・速度が主目的

---

## 11. リポジトリ運用方針

v1（Python）と v2（Rust）を同居させる場合、モノレポ構成を採用する。

### 構成案
- `python/` … v1.x（安定版）
- `rust/` … v2.x（開発版）
- `docs/` … 設計メモ・ロードマップ

### タグ運用（推奨）
混同防止のため、prefix 付きタグも検討する。
例：
- `py-v1.0.1`
- `rs-v2.0.0-alpha1`

※ シンプルに `v1.x` / `v2.x` でも運用可能だが、モノレポでは prefix 付きが安全
